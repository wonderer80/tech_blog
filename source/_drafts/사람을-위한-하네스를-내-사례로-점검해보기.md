[1편](/2026/04/25/사람을-위한-하네스/)에서는 AI 에이전트 하네스의 12요소를 사람에게 옮겨 표로 비추어 보고, 그 표를 자기 일에 비추어 빈자리를 점검해보면 좋겠다고 적었다. [2편](/2026/04/26/사람을-위한-하네스를-만들고-키우기/)에서는 그 12요소를 한 칸씩 풀어 본 뒤, 사람을 위한 하네스를 어떻게 만들고 키워갈지를 다뤘다.

이번 글은 그 표를 실제로 내 일에 비추어 점검해본 결과다. 본문이라기보다 점검 노트에 가깝고, 안에 등장하는 도구·시스템 이름은 내가 평소 쓰는 것들이라 다른 분들에게는 낯설 수 있다. 그래도 한 사람이 12칸을 어떻게 비추어 보는지의 한 예시로 남겨둔다.

## 12칸 점검 결과

| 층 | 사람을 위한 하네스 항목 | 현재 내가 가진 것 | 상태 | 비어있거나 부족한 부분 |
|---|---|---|---|---|
| 런타임 | 하루의 사이클 (계획 → 실행 → 기록 → 다음) | 데일리 스크럼, 오늘 원씽, Akiflow 오늘/예정, worklog, daily memory | 강함 | 컨디션 저하나 일정 꼬임 때 루프를 자동 축소하는 장치는 약함 |
| 런타임 | 작고 분명한 다음 행동, 입력과 기대 결과의 명시 | task-coach식 쪼개기, 원씽, 10분짜리 작업으로 변환 | 중간 이상 | 막막한 일에 항상 자동 적용되지는 않음 |
| 런타임 | 출력 파싱 | 데일리 스크럼 요약, worklog 정리, 로미와의 대화 요약 | 약함 ~ 부분적 | 회의/문서/생각을 결정/액션 아이템/보류로 일관되게 추출하는 구조가 약함 |
| 런타임 | 오류 처리 | `memory/lessons.md`, worklog, git commit, 설정 변경 전 승인, 재발 방지 기록 | 강함 | 생활/개인 시스템 쪽은 코드만큼 되돌리기 쉬운 단위가 항상 있지는 않음 |
| 기능 | 도구 | Akiflow, Google Calendar, worklog, todo.md, memory, OpenClaw, 로미, Claude Code, Codex, 자동화 스크립트 | 강함 | 도구 수는 충분하지만 무엇을 어디서 처리할지 경계가 흐려질 때가 있음 |
| 기능 | 메모리 | daily memory, `MEMORY.md`, 회사 `worklog.md`, `todo.md`, 볼링/다이어트 로그 | 아주 강함 | 메모는 많지만 장기적으로 자주 다시 보는 핵심 운영 문서는 더 압축될 여지가 있음 |
| 기능 | 컨텍스트 관리 | 데일리 스크럼, 오늘 원씽, worklog, 진행 메모, 로미의 문맥 이어받기 | 꽤 강함 | 낮은 에너지 상태에서 어떤 버전으로 축소할지에 대한 규칙은 약함 |
| 기능 | 상태 관리 | Akiflow today/inbox/planned, 회사 todo.md, GitHub 이슈/PR, 진행중/대기중 구분 | 꽤 강함 | 활성 작업 수를 강하게 제한하는 장치는 약함 |
| 안전과 확장 | 가드레일과 안전 | 시스템 설정 변경 전 승인, 위험 작업 전 확인, 코치봇/로미 가드레일, PR/리뷰 문화 | 중간 이상 | 개인 의사결정이나 과부하 상태에서 무리한 계획을 막는 체크리스트는 약함 |
| 안전과 확장 | 검증 루프 | PR 리뷰, 팀 공유, 모니터링, 로미와 회고/분석, 코치 계열 스킬 | 강해지는 중 | 개인 할 일/생활 쪽은 검증이 늦고, 시스템 성능 평가 기준이 아직 약함 |
| 안전과 확장 | 하위 에이전트 오케스트레이션 | Claude Code, Codex, 로미, 코치봇, 자동화 스크립트, heartbeat, cron | 강함 | 위임 결과의 품질 검수 기준이 더 명시적이면 좋음 |
| 안전과 확장 | 도구 범위 전략 | 일부 heartbeat/cron 분리, 메일 정리/주간 태스크 정리 같은 배치 처리, 원씽 개념 | 약함 | 시간대별/컨디션별/작업 종류별로 도구를 분리하는 전략이 약함 |

## 강점부터 정리하면

- 기억 장치가 강하다.
- 복구 장치가 강하다.
- 위임 장치가 강하다.
- 기록 습관이 강하다.

## 약점부터 정리하면

- 선택지 제한 장치가 약하다.
- 출력 파싱 장치가 약하다.
- 에너지·컨디션 대응 장치가 약하다.
- 도구 범위 전략이 약하다.
- 하네스 자체의 성능을 재는 기준이 아직 약하다.

## 먼저 보강할 후보

이 약점들 중에서 우선순위를 정해본다면 다음과 같다.

1. **선택지 제한 장치** — 후보가 너무 많아질 때 자동으로 줄여주는 장치가 더 필요하다.
2. **출력 파싱 장치** — 회의·문서·생각을 바로 결정과 액션 아이템으로 바꾸는 파이프라인이 필요하다.
3. **에너지 기반 운영 규칙** — 컨디션이 낮은 날 계획을 자동 축소하는 규칙이 필요하다.
4. **하네스 신뢰성** — 자동화가 가끔 실패하므로 반자동 확인 루프가 더 명시적이면 좋다.
5. **도구 범위 전략** — 시간대별, 작업 종류별, 집중도별로 도구를 더 분리할 필요가 있다.

이 다섯 가지를 한꺼번에 손볼 생각은 없다. 가장 자주 무너지는 한 칸부터 손보고, 다음 주에 다시 12칸을 비추어 본다. 빈자리는 한 번에 채우려 하지 않는다. 한 칸씩, 가장 자주 무너지는 곳부터 손본다는 것이 1편에서 적어둔 약속이었다.
